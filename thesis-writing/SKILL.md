# 本科毕业论文写作辅助

## 适用场景

- 用户要求规划论文章节结构或生成大纲
- 用户要求润色段落、改写为学术表达
- 用户要求生成中文摘要或英文 Abstract
- 用户要求格式化参考文献（GB/T 7714）
- 用户要求撰写图表标题与描述
- 用户要求检查论文格式是否符合学校规范
- 用户要求评估论文质量、给出改进建议

## 基本约束

- **语言**：中文为主，英文 Abstract 需与中文摘要内容对应
- **学科**：计算机科学 / 人工智能方向
- **学位层次**：本科毕业论文（设计）
- **格式规范**：严格遵循 [绵阳城市学院撰写规范](./references/formatting-spec.md)
- **输出工具**：Microsoft 365 Word

## 读取 Word 文档

当用户提供 `.docx` 文件路径时，使用 [read_docx.py](./scripts/read_docx.py) 脚本提取内容或格式信息。**必须使用 `--output` 参数将结果写入临时文件**，避免 Windows 终端编码问题，然后用 `read_file` 工具读取该文件：

```bash
# 提取文本内容（用于润色、摘要生成、综述写作等）
python "~/.copilot/skills/thesis-writing/scripts/read_docx.py" "<文件路径>" --mode content -o "$env:TEMP/docx_output.txt"

# 提取格式信息（用于格式合规检查）
python "~/.copilot/skills/thesis-writing/scripts/read_docx.py" "<文件路径>" --mode format -o "$env:TEMP/docx_output.txt"

# 同时提取内容和格式
python "~/.copilot/skills/thesis-writing/scripts/read_docx.py" "<文件路径>" --mode all -o "$env:TEMP/docx_output.txt"
```

运行后使用 `read_file` 读取 `$env:TEMP/docx_output.txt` 获取结果（UTF-8 编码，无乱码）。

首次使用前需安装依赖：`pip install python-docx`

脚本参数说明：
- `--mode content|format|all`：提取模式（默认 all）
- `--output / -o`：输出文件路径（**推荐使用**，避免终端编码乱码）

脚本输出信息包括：
- **content 模式**：带标题层级标记的结构化文本、表格内容、字数统计
- **format 模式**：页面设置（边距/页眉/页脚）、每种样式的字体字号、行距、段距、对齐方式等

## 工作流程

根据用户请求的类型，执行对应流程：

---

### 流程 A：章节结构规划与大纲生成

1. **确认论文选题与研究方向**：向用户了解论文题目、研究问题、方法和主要贡献。
2. **生成标准结构大纲**，必须包含以下完整部分：
   - 摘要（中文 + 英文）
   - 目录
   - 第 1 章 绪论（研究背景与目的、国内外研究现状/文献综述、研究方法与内容）
   - 第 2~N 章 主体（根据研究内容拆分，每章聚焦一个核心问题）
   - 结论（无章节号）
   - 致谢（无章节号）
   - 参考文献（无章节号）
   - 附录（必要时，无章节号）
3. **层次编号**采用理工类格式：`1` → `1.1` → `1.1.1` → `（1）`
4. **每章给出简要说明**（2~3 句），概括该章要解决的问题和预期内容。
5. **工作量提示**：正文不少于 9000 字；参考文献不少于 20 篇（期刊 ≥10、外文 ≥3、近 3 年 ≥6）。

---

### 流程 B：段落润色与学术表达改写

1. **接收用户提供的原始段落**。
2. **改写原则**：
   - 使用"本文""本系统""该系统""该模块""该方法"等作为主语，避免"我""我们"（除致谢外）。
   - 采用客观、严谨、简练的学术语言，避免口语化表达。
   - 术语统一：全文同一概念使用同一术语，英文缩写首次出现时括注全称。
   - 逻辑连接词恰当使用（因此、然而、此外、综上所述等）。
   - 数据描述准确，量和单位符合国标 GB3100~GB3102。
3. **输出改写后的段落**，并用简短注释说明主要修改点。
4. **不改变原文的技术含义和核心观点**。
5. **质量自检**：润色完成后，参照 [error-patterns.md](./references/error-patterns.md) 中 B 类"写作中检查清单"，确认无口语化（#18）、主语规范（#20）、术语统一（#15）等问题。

---

### 流程 C：摘要 / Abstract 生成

1. **输入**：用户提供论文的研究背景、方法、主要结果和结论（或提供完整论文/大纲）。
2. **中文摘要**：
   - 约 300 字，涵盖：研究目的、方法、结果、结论。重点突出结果和结论。
   - 不使用图、表、非公知符号。
   - 末尾另起一行，列出 3~8 个关键词，关键词之间用"；"和一个空格隔开，最后一个关键词后不加标点。
3. **英文 Abstract**：
   - 与中文摘要内容严格对应。
   - 使用 Times New Roman 体的规范学术英语，语法正确、表述清晰。
   - Key words 之间用", "隔开，最后一个不加标点。
4. **输出格式**：分别标注"摘要""关键词""Abstract""Key words"。
5. **质量自检**：生成后参照 [error-patterns.md](./references/error-patterns.md) 中 D 类"全文后检查清单"，确认摘要四要素完整、关键词准确。

---

### 流程 D：参考文献格式化

1. **输入**：用户提供文献信息（标题、作者、期刊/出版社、年份等），或未格式化的文献列表。
2. **严格按照 GB/T 7714-2005 格式化**，类型标识规则：
   | 类型 | 标识 |
   |------|------|
   | 期刊文章 | [J] |
   | 普通图书 | [M] |
   | 论文集 | [C] |
   | 学位论文 | [D] |
   | 专利 | [P] |
   | 标准 | [S] |
   | 报纸文章 | [N] |
   | 报告 | [R] |
   | 电子文献 | [文献类型/载体类型] |
3. **著录格式模板**（详见 [formatting-spec.md](./references/formatting-spec.md) 第 3.15 节）：
   - 期刊：`[序号] 作者.文献题名[J].刊名,出版年份,卷号(期号):起-止页码.`
   - 著作：`[序号] 作者.书名[M].版次.翻译者.出版地:出版社,出版年:起-止页码.`
   - 学位论文：`[序号] 作者.题名[D].保存地:保存单位,年份.`
   - 其他类型参照规范。
4. **序号按引用顺序编排**，左顶格，`[1]`、`[2]`……每条以 `.` 结束。
5. **外文文献保持原文**，不翻译成中文。

---

### 流程 E：图表标题与描述撰写

1. **图题**：
   - 格式：`图 X-Y 图名`（按章编号，如"图 2-1 系统架构图"）。
   - 置于图下居中，5 号宋体加粗，数字和字母为 5 号 Times New Roman 加粗。
   - 图名简短准确，不使用标点。
   - 分图编号用 (a)、(b)、(c) 等。
2. **表题**：
   - 格式：`表 X-Y 表名`（按章编号，如"表 3-1 实验结果对比"）。
   - 置于表上居中，5 号宋体加粗，数字和字母为 5 号 Times New Roman 加粗。
   - 表名不使用标点、表名后不加标点。
   - 三线表格式：顶线和底线 1.5 磅，栏目线 0.5 磅。
3. **正文引用**：在文中用"如图 X-Y 所示""如表 X-Y 所示"引出。
4. 如果用户提供了数据或图片内容，帮助撰写对图表内容的描述性段落。

---

### 流程 G：文献综述写作

1. **输入**：用户提供若干篇参考文献（标题、摘要或关键结论），以及综述所在章节的主题。
2. **分析与分类**：
   - 按研究方向、方法或时间线对文献进行逻辑分组。
   - 识别各组文献的共同点、差异和研究空白。
3. **组织综述段落**：
   - 每组文献形成一个或多个段落，段落间有逻辑过渡。
   - 采用"总—分—总"结构：先概述该方向整体情况，再分述代表性工作，最后总结不足或趋势。
   - 引用标注使用顺序编码制上标 `[1]`，序号与用户提供的文献编号对应。
4. **写作要求**：
   - 客观评述，不照搬原文表述，用自己的语言归纳。
   - 突出与本文研究的关联：指出已有工作的局限，自然引出本文的研究动机。
   - 每段末尾或段间体现批判性思考（如"然而……尚未解决""但存在……不足"）。
5. **输出**：格式化的综述段落，引用标注完整，可直接嵌入论文正文。
6. **质量自检**：参照 [error-patterns.md](./references/error-patterns.md) 中错误模式 #21（文献综述变文献列举），确认综述段落具有归纳、对比和批判性分析，而非逐篇摘要。

---

### 流程 F：格式合规检查

当用户提供 `.docx` 文件路径时，先运行 `read_docx.py --mode format` 提取实际格式，然后对照 [格式规范](./references/formatting-spec.md) 逐项核对，输出检查清单：

1. **结构完整性**：封面 → 诚信声明 → 摘要 → 目录 → 正文 → 致谢 → 参考文献 → 附录
2. **字体字号**：论文题目（小二黑体）、一级标题（小二黑体）、二级标题（四号黑体）、三级标题（小四宋体）、正文（小四宋体）、参考文献内容（五号宋体）等
3. **页面设置**：上/下/左边距 3.0cm、右边距 2.5cm，页眉 2.0cm、页脚 2.0cm
4. **页眉**：内容为"绵阳城市学院本科生毕业论文"，五号华文行楷
5. **页码**：正文起阿拉伯数字连续编排，摘要和目录用罗马数字
6. **行距**：全文无网格、固定值 22 磅，段前段后不空行
7. **标题样式**：一级标题另起一页居中（段前空 1 行/段后空 0.5 行），理工类二级标题左起空两字符（段前段后各空 0.25 行）
8. **引用标注**：顺序编码制，上标方括号 `[1]`
9. **参考文献**：GB/T 7714 格式，数量达标（≥20 篇）
10. **图表规范**：编号按章、标题位置正确、三线表格式
11. **工作量**：正文 ≥9000 字

对每一项输出 ✅（合格）或 ❌（不合格 + 具体问题 + 修改建议）。

---

### 流程 H：论文质量评估与改进建议

当用户提供 `.docx` 文件路径或粘贴论文内容，要求评估、审阅或给建议时，执行此流程。如有 `.docx` 文件，先运行 `read_docx.py --mode content` 提取全文。

本流程采用**三阶段审阅**方法论（改编自 [thesis-harness](https://github.com/AlbanLi0314/thesis-harness)），结合 **7 维度评分**，确保评估全面。详细的错误模式库和检查清单见 [error-patterns.md](./references/error-patterns.md)。

#### 第一阶段：独立冷读（发现内容与逻辑问题）

以"第一次读到这篇论文的陌生读者"视角通读全文，**不**参考写作计划或用户说明。逐段落追问：
- 这段想表达什么？表达成功了吗？
- 逻辑链条是否完整？有无跳跃？
- 有没有"读起来很顺畅但仔细想不对"的论断？（叙事完美偏差）
- 不了解项目背景的读者能理解吗？

记录为编号列表（IR-1, IR-2...），标注严重程度（高/中/低）。

#### 第二阶段：规范对照（发现格式与规范问题）

以"学校论文审查员"视角，对照 [格式规范](./references/formatting-spec.md) 和 [错误模式库](./references/error-patterns.md) 检查。如有 `.docx` 文件，可配合 `read_docx.py --mode format`。

记录为编号列表（SC-1, SC-2...），与第一阶段合并。

#### 第三阶段：答辩委员视角（发现深层学术问题）

以"答辩评委提问"的视角重读论文，对每个核心论断追问：
- **"凭什么？"** — 证据是否充分？
- **"所以呢？"** — 这部分的贡献是什么？还是只在报告数据？
- **"你和别人有什么不同？"** — 创新声明是否有支撑？
- **"那另一种解释呢？"** — 是否忽略了反面论据？
- **"你只测了 N 个条件——怎么得出这个结论？"** — 局限性是否被承认？
- **"前面说 X，后面说 Y——到底是哪个？"** — 内部一致性。

记录为编号列表（CM-1, CM-2...），与前两阶段合并。

#### 7 维度评分

三阶段完成后，将所有发现归入以下 7 个维度并评级：

**维度 1：结构完整性与逻辑性**
- 各章节是否齐全、逻辑链条是否清晰
- 各章篇幅是否均衡，"本章小结"是否真正有内容
- 绪论提出的问题在后续章节是否都有回应
- 错误模式参考：#1, #2, #3

**维度 2：绪论与文献综述质量**
- 研究背景是否建立了问题的必要性和紧迫性
- 文献综述是"综述"（归纳、对比、评价）还是"列举"（逐篇摘要）→ 错误模式 #21
- 国内外现状是否自然引出本文的研究动机
- 参考文献数量/质量是否达标

**维度 3：技术方案与创新性**
- 方案选型是否有对比论证 → 错误模式 #22
- 技术路线是否完整可行，关键参数有无数据支撑
- 创新点是否明确、有说服力
- 是否存在"声称创新但无数据支撑"的问题

**维度 4：实验/测试与结果分析**
- 测试方案是否科学（有对照、可重复、样本量合理）
- 数据是否充分，有无图表展示
- 结果是否有深入分析 → 错误模式 #23
- 是否讨论了局限性和不足

**维度 5：写作质量与学术规范**
- 语言是否学术化（无口语 → #18、无主观臆断）
- 主语是否规范 → 错误模式 #20
- 术语是否全文统一 → 错误模式 #15
- 有无大段重复、冗余或前后矛盾

**维度 6：图表与参考文献规范**
- 图表编号、标题是否完整，正文是否引用 → 错误模式 #26
- 参考文献格式是否符合 GB/T 7714
- 有无"僵尸文献"或"幽灵引用" → 错误模式 #25
- 引用标注格式是否正确（顺序编码制上标）

**维度 7：摘要与结论质量**
- 摘要是否涵盖目的、方法、结果、结论四要素
- 英文 Abstract 是否与中文对应、语法正确
- 结论是否精练总结创新成果 → 错误模式 #24
- 是否包含局限性讨论和未来展望

#### 输出格式

对每个维度输出：

```
### 维度 N：XXX
**评级**：⭐⭐⭐☆☆（3/5）
**亮点**：简述做得好的地方
**问题**：列出具体问题，引用原文位置和对应的错误模式编号
**建议**：给出可操作的改进方案
```

最后给出：
- **总体评价**：综合评分 + 一句概括
- **优先修改清单**：按严重程度排序的 Top 5~10 待改项，格式如下：

| # | 来源 | 位置 | 问题 | 严重度 | 修改方案 |
|---|------|------|------|--------|---------|
| 1 | IR-3 | 第2章§2.1 | 选型无对比论证（#22） | 高 | 补充方案对比表 |

- 先处理所有"高"级问题，再处理"中"级
- 多个阶段指向同一段落 → 该段落需要重写，而非修补

## 写作风格指引

- **主语偏好**：使用"本文""本系统""该系统""该模块""该方法"等客观主语，避免"我""我们"（致谢除外）。
- **客观严谨**：避免主观臆断，论点有据可查。
- **逻辑清晰**：段落之间有逻辑过渡，章节之间有承上启下。
- **术语规范**：采用国标术语，英文缩写首次出现括注全称。
- **标点正确**：遵循新闻出版署《标点符号用法》。
- **数字使用**：统计数据用阿拉伯数字，叙述性小数目用中文数字。
