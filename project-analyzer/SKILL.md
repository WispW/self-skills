# 项目分析评估技能

## 使用场景

当用户要求以下操作时，必须使用此skill：
- 分析项目
- 评估项目质量
- 项目诊断
- 代码审查评估
- 项目健康度检查
- 获取项目改进建议
- 项目现状分析

## 核心分析维度

### 1. 代码质量维度
- **代码结构**：目录组织、模块划分、命名规范
- **代码复杂度**：循环复杂度、函数长度、圈复杂度
- **代码重复**：重复代码检测、代码克隆
- **代码规范**：缩进、注释、文档、代码风格一致性
- **坏味道检测**：过长函数、过多参数、长参数列表等

### 2. 技术栈分析维度
- **主要语言**：检测项目使用的编程语言及版本
- **框架识别**：识别使用的框架和库
- **依赖分析**：依赖版本、过时依赖、漏洞依赖
- **构建系统**：构建工具、构建配置、打包方式
- **技术组合**：技术栈的现代性评估

### 3. 安全性维度
- **依赖安全**：检测已知漏洞、版本风险
- **代码安全问题**：硬编码密钥、SQL注入风险、XSS风险等
- **配置安全**：敏感信息暴露、默认密码、权限配置
- **认证授权**：访问控制实现、密码策略
- **加密通信**：HTTPS使用、加密存储

### 4. 性能维度
- **资源优化**：包体积、依赖大小、构建产物优化
- **运行效率**：关键路径分析、算法复杂度
- **缓存策略**：缓存使用、内存管理
- **并发处理**：并发模型、锁使用、线程安全
- **性能基线**：是否有性能测试

### 5. 项目配置维度
- **构建配置**：pom.xml、package.json、build.gradle等
- **环境配置**：环境变量管理、多环境配置
- **CI/CD**：是否有持续集成、持续部署配置
- **测试配置**：测试框架、测试覆盖率
- **文档配置**：README、API文档、设计文档

### 6. 可维护性维度
- **文档完整性**：README、API文档、架构文档、注释
- **测试覆盖**：单元测试、集成测试、覆盖率
- **模块化程度**：耦合度、内聚度、模块独立性
- **版本管理**：版本号规范、CHANGELOG维护
- **日志规范**：日志级别使用、日志输出规范

## 分析工作流程

### 第一阶段：项目识别
1. 扫描项目根目录，识别所有配置文件
2. 确定项目主要编程语言
3. 识别项目类型（Web、CLI、库、微服务等）
4. 识别使用的框架和构建系统

### 第二阶段：数据收集
1. **结构分析**：遍历项目目录，分析代码组织
2. **依赖分析**：解析package.json、pom.xml、requirements.txt等依赖文件
3. **代码分析**：统计代码行数、文件数量、函数数量等指标
4. **配置分析**：检查各类配置文件的完整性和规范性
5. **文件清单**：列出关键目录和文件

### 第三阶段：深度评估
对每个维度进行评分：
- 代码质量评分（0-100分）
- 技术栈现代性评分（0-100分）
- 安全性评分（0-100分）
- 性能基线评分（0-100分）
- 可维护性评分（0-100分）

### 第四阶段：问题识别和优先级分类

#### 优先级评分算法
```
优先级分数 = 影响程度(0-40) + 出现频率(0-30) + 解决难度权重(0-30)

- 影响程度：问题对系统的实际影响
  - 严重(40分)：影响安全、功能、性能
  - 中等(25分)：影响代码质量、可维护性
  - 轻微(10分)：代码风格、文档完善
  
- 出现频率：问题出现的次数
  - 频繁(30分)：5次以上
  - 多次(20分)：2-4次
  - 偶现(10分)：1次
  
- 解决难度（反向：越难分数越高）
  - 容易(10分)：立即可修复
  - 中等(20分)：需要重构或代码变更
  - 困难(30分)：需要架构调整或大规模改动
```

#### 优先级分类
- **🔴 严重(Critical, 76-100分)**：必须立即处理
- **🟠 高(High, 51-75分)**：应该尽快处理
- **🟡 中(Medium, 26-50分)**：计划处理
- **🟢 低(Low, 0-25分)**：可选优化

# 项目分析评估报告：API Gateway Service

| 问题 | 位置 | 影响 | 建议 |
|-----|------|------|------|
| 依赖中存在已知漏洞（高危） | package.json | 安全漏洞 | 升级 express 至最新版本，运行 npm audit fix |

## 实施要点

### 分析时应该考虑的工具

根据项目类型，可以参考以下工具：

**JavaScript/Node.js项目：**
- npm audit（依赖漏洞检测）
- ESLint（代码规范检查）
- SonarQube（代码质量分析）
- Jest（测试框架）

**Python项目：**
- pip-audit（依赖漏洞检测）
- pylint（代码规范检查）
- SonarQube（代码质量分析）
- pytest（测试框架）

**Java项目：**
- OWASP Dependency-Check（依赖漏洞检测）
- Checkstyle（代码规范检查）
- SonarQube（代码质量分析）
- JUnit（测试框架）

### 分析的关键问题清单

#### 代码质量问题
- 是否存在过长函数（>200行）？
- 是否存在高复杂度函数（圈复杂度>10）？
- 是否存在重复代码？
- 代码注释率是否足够（>20%）？
- 是否遵循命名规范？

#### 安全性问题
- 是否存在已知漏洞依赖？
- 是否有硬编码的敏感信息？
- API是否有身份验证和授权？
- 是否有CSRF、XSS保护？
- 数据库是否有SQL注入保护？

#### 性能问题
- 是否存在N+1查询问题？
- 是否有缓存策略？
- 包体积是否合理？
- 数据库查询是否有优化空间？

#### 可维护性问题
- 是否有完整的文档（README、API文档）？
- 测试覆盖率是否足够（目标>70%）？
- 是否有CI/CD流程？
- 版本管理是否规范？

## 使用建议

1. **首次分析**：进行完整的五维度分析，了解项目现状
2. **定期复查**：建议每月进行一次完整分析，跟踪改进情况
3. **持续优化**：根据优先级分类，逐步解决问题
4. **团队协作**：将分析结果分享给团队，制定改进计划

## 输出格式规范

所有分析报告应遵循以下格式：

1. 使用Markdown格式编写
2. 包含清晰的目录结构
3. 使用表格展示问题清单
4. 优先级用颜色emoji标注（🔴 🟠 🟡 🟢）
5. 提供具体的改进建议和行动计划
6. 包含代码示例或引用具体的文件位置和行号
